## Critical Issues (app crashes, data loss, security)

### [C1] generate-image route: Product.imageUrl does not exist in schema
- **File:** `src/app/api/ai/generate-image/route.ts:61,79`
- **Description:** Route writes `imageUrl` to Product model, but the field doesn't exist in `prisma/schema.prisma`. Every call throws a Prisma runtime error.
- **Impact:** Route is completely broken. Will 500 on every request.
- **Fix:** Either add `imageUrl String?` to Product model, or remove/disable the route.

### [C2] generate-image route: No authentication
- **File:** `src/app/api/ai/generate-image/route.ts`
- **Description:** No `requireAuth()` call. Any unauthenticated user can trigger image generation (costs money via Gemini API + writes to storage).
- **Impact:** Security vulnerability — unauthenticated access to a paid API.
- **Fix:** Add `requireAuth()` at top of handler.

### [C3] test/reset-receiving route: No authentication
- **File:** `src/app/api/test/reset-receiving/route.ts`
- **Description:** Only checks `NODE_ENV === "production"` but has no auth. In dev/staging, any request can wipe all receipts, transactions, and reset all product quantities to 0.
- **Impact:** Data destruction in non-production environments.
- **Fix:** Add `requireAuth()` + `requireRole(["ADMIN"])`.

## Major Issues (broken flows, wrong data, bad UX)

### [M1] CheckoutAllButton ignores qtyReturned — miscalculates remaining
- **File:** `src/components/bom/checkout-all-button.tsx`
- **Description:** `remaining = qtyNeeded - qtyCheckedOut` does not account for `qtyReturned`. If a user checked out 10, returned 5, the button thinks 10 are checked out and shows 0 remaining, when 5 should be available to re-checkout.
- **Impact:** Wrong remaining quantities shown, checkout button may be hidden when items are actually available.

### [M2] Receiving flow: second photo parse overwrites first
- **File:** `src/components/receiving/receiving-flow.tsx:89`
- **Description:** `setPendingMatches(result.items)` replaces rather than merges. If a user snaps two photos before confirming, the second parse completely overwrites the first batch.
- **Impact:** Data loss — first photo's parsed items silently disappear.

### [M3] Shipping: silent data fetch failure shows misleading empty state
- **File:** `src/components/shipping/finished-goods-list.tsx`
- **Description:** `completedQuery.error` and `shippedQuery.error` are never checked. If API fails, `completed` defaults to `[]` and user sees "All Shipped" instead of an error.
- **Impact:** User thinks all items are shipped when the API actually failed.

### [M4] Stock management: race condition on concurrent adjustStock
- **File:** `src/lib/stock.ts`
- **Description:** Reads `product.currentQty`, computes `newQty` in JS, then writes back. Under PostgreSQL's default `READ COMMITTED` isolation, two concurrent calls can interleave and last-write-wins (lost update).
- **Impact:** Inventory quantity could be incorrect under concurrent operations (e.g., two people checking out simultaneously).
- **Fix:** Use `prisma.product.update({ data: { currentQty: { increment: qty } } })` for atomic updates.

### [M5] BOM detail: no back navigation
- **File:** `src/app/boms/[id]/page.tsx`
- **Description:** Header uses `showMenu` but not `showBack`. No way to navigate back to BOM list except browser back.
- **Impact:** Dead end for users who deep-linked or lost their history stack.

### [M6] Receiving: SUMMARY phase has no back button
- **File:** `src/components/receiving/receiving-flow.tsx`
- **Description:** SUMMARY phase has no back or "Start over" button. User can only escape via the step progress breadcrumb tap, which is not discoverable.
- **Impact:** User gets stuck if they realize something is wrong at the summary step.

### [M7] Receiving: PO match failures silently swallowed
- **File:** `src/components/receiving/receiving-flow.tsx:121,143`
- **Description:** Catch blocks are empty with only comments. No toast or user feedback when PO matching fails.
- **Impact:** User doesn't know PO matching failed — proceeds without PO context.

### [M8] Assembly: ALLOCATED status is a dead end
- **File:** `src/app/assemblies/[id]/page.tsx`
- **Description:** The detail page has action buttons for APPROVED, IN_PRODUCTION, COMPLETED, and SHIPPED — but nothing for ALLOCATED. User viewing an ALLOCATED assembly has no way to move it forward.
- **Impact:** Dead-end state — assembly stuck in ALLOCATED with no actions.

### [M9] Door builder: dimension inputs accept empty strings
- **File:** `src/components/doors/door-builder.tsx`
- **Description:** `DimensionInput` component does not validate that input is non-empty. User can advance through width, height, jamb depth with completely empty values.
- **Impact:** Door created with no dimensions — invalid spec.

## Suggestions (not bugs, but improvements)

### [S1] Use atomic increment/decrement in stock.ts
Replace `read → compute → write` pattern with `prisma.product.update({ data: { currentQty: { increment: qty } } })` for atomic updates. Eliminates the race condition entirely.

### [S2] Add Zod validation to AI parse routes
`ai/parse`, `ai/refine-matches`, `suppliers/match`, `unit-conversions` POST all lack Zod schemas. Adding runtime validation prevents malformed input from reaching business logic.

### [S3] Standardize getCurrentUser vs requireAuth
6 routes use `getCurrentUser()` + manual null check instead of `requireAuth()`. Standardizing to `requireAuth()` reduces inconsistency.

### [S4] Add negative stock guard or warning
`stock.ts` allows negative quantities with no guard. Consider either preventing negative stock or logging a warning when it occurs.

### [S5] Fix ALL_COLOR_NAMES.sort() mutation
`panels.ts` line 159 mutates a module-level array on every call. Use `[...ALL_COLOR_NAMES].sort()` instead.
