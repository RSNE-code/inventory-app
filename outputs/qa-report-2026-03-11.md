## Medium Issues

### [M1] Hardcoded `unitOfMeasure: "each"` in PO Receive Card
- **File:** `src/components/receiving/po-receive-card.tsx:46`
- **Description:** When building `ConfirmedReceivingItem` from PO line items, unit of measure is hardcoded to `"each"` instead of using the product's actual UOM from the PO line item or matched product.
- **Impact:** Products sold by linear ft, sq ft, sheet, etc. will be received with incorrect UOM, potentially causing stock calculation errors downstream.

### [M2] PO Search API errors silently swallowed
- **File:** `src/components/receiving/po-match-card.tsx:293`
- **Description:** `usePoSearch` hook errors are not displayed to the user. If the API call fails, the search view just shows "No open POs" with no error indication.
- **Impact:** Users won't know if search failed vs. no results found.

### [M3] Supplier picker API errors silently swallowed
- **File:** `src/components/receiving/supplier-picker.tsx` (if used in flow)
- **Description:** Similar to M2 — API errors in supplier search are not surfaced to the user.
- **Impact:** Silent failures during supplier matching.

### [M4] Duplicate `rawText` keys can cause items to disappear
- **File:** `src/components/receiving/receiving-flow.tsx` (item keying)
- **Description:** Items in the review/confirmation list are keyed by `rawText`. If two AI-parsed items have identical `rawText` values (e.g., two line items with the same description), React will only render one of them.
- **Impact:** Items silently dropped from the receiving list when duplicates exist.

### [M5] Unbounded PO context loading in AI parse
- **File:** `src/lib/ai/parse.ts:47-67`
- **Description:** `loadPOContext()` loads ALL purchase orders with ALL line items, regardless of count. As PO volume grows, this will bloat the AI prompt token count significantly, increasing cost and potentially hitting context limits.
- **Impact:** Increasing AI costs and eventual failures as PO count grows over time.

### [M6] Non-existent `poLineItemId` causes opaque 500 error
- **File:** `src/app/api/receiving/route.ts`
- **Description:** If a `poLineItemId` in the `poLineItemUpdates` array doesn't exist in the database, Prisma will throw a generic error. There's no pre-validation that the IDs exist.
- **Impact:** Opaque 500 error instead of a clear "line item not found" message.

## Minor Issues

### [m1] Middleware deprecation warning
- **File:** `src/middleware.ts`
- **Description:** Next.js 16 shows a deprecation warning for `middleware.ts` in favor of the new `proxy` system. Functionally still works.
- **Impact:** Console warning during build, no functional impact.

### [m2] `usePoSearch` double `.data` access pattern
- **File:** `src/components/receiving/po-match-card.tsx:294`
- **Description:** The hook returns `{ data }` which itself contains `{ data: [...] }`, requiring `data?.data ?? []` access pattern. While functional, it's an inconsistent API shape.
- **Impact:** Confusing for maintenance, but works correctly.

### [m3] PO `RECEIVED` status defined but never used
- **File:** `prisma/schema.prisma` (POStatus enum)
- **Description:** The `RECEIVED` status exists in the enum but the receiving API only transitions POs to `PARTIALLY_RECEIVED` or `CLOSED`. `RECEIVED` is never set by any code path.
- **Impact:** Dead enum value, no functional impact.

### [m4] Partial PO matching can produce false positives
- **File:** `src/lib/ai/po-match.ts` (if exists) / `src/lib/ai/parse.ts`
- **Description:** When the AI can't find an exact PO number match, it falls back to fuzzy matching by supplier + items. This can match the wrong PO when a supplier has multiple similar orders.
- **Impact:** Users may be presented with an incorrect PO match, though they can reject it and search manually.

---

## Suggestions

### [S1] Add PO context pagination/filtering
- Limit `loadPOContext()` to only OPEN and PARTIALLY_RECEIVED POs, and optionally limit to last 90 days. This would reduce token usage significantly as PO volume grows.

### [S2] Add `unitOfMeasure` to PO line item data
- Either store UOM on `POLineItem` records or resolve it from the matched product when building the receiving checklist, so PO-based receiving uses accurate units.
