## Critical Issues (app crashes, data loss, security)

### [C1] AI Parse API Routes Have No Error Handling
- **Files:** `src/app/api/ai/parse/route.ts`, `src/app/api/ai/parse-image/route.ts`
- **Description:** These are the only two API routes with no try/catch. If the Anthropic API is down, the AI returns non-JSON, or catalog matching fails, the error propagates as an unhandled 500 with a potential stack trace leak.
- **Impact:** User sees a generic server error with no useful message. Stack trace could expose internal paths.
- **Reproduction:** Send any request when ANTHROPIC_API_KEY is invalid or Anthropic API is unreachable.

### [C2] AI Response Structure Not Validated After Parsing
- **File:** `src/lib/ai/parse.ts:78,112`
- **Description:** After `extractJSON()`, the result is cast as `{ items: ParsedLineItem[] }` without validation. If the AI returns `{"results": [...]}` instead of `{"items": [...]}`, `parsed.items` is `undefined`, causing a crash in `matchItemsToCatalog()` when it tries to `.map()` on `undefined`.
- **Impact:** Runtime crash on malformed AI response — entire receiving/BOM/checkout flow breaks.
- **Reproduction:** AI model returns valid JSON with unexpected structure.

### [C3] Jobs API Routes Have No Role Checks
- **Files:** `src/app/api/jobs/route.ts:58` (POST), `src/app/api/jobs/[id]/route.ts:34` (PUT)
- **Description:** Both routes call `requireAuth()` but NOT `requireRole()`. Any authenticated user (including CREW role) can create and edit jobs. Every other write endpoint restricts by role.
- **Impact:** Unauthorized users can create/modify job records.
- **Reproduction:** Log in as CREW user, POST to `/api/jobs`.

### [C4] Cycle Count "Reason" Field Never Saved
- **File:** `src/app/cycle-counts/page.tsx:199-205`
- **Description:** The "Reason" text input appears AFTER the count is submitted. `handleSubmitCount` runs first (line 66-79, sends `reason: reason.trim() || null`), but the reason input is only visible in the result screen. Since `reason` state is always `""` at submission time, variance reasons are never recorded.
- **Impact:** All cycle count variances are saved with null reason — audit trail is incomplete.
- **Reproduction:** Do any cycle count with a variance.

## Major Issues (broken flows, wrong data, bad UX)

### [M1] Door Builder: `finish` Field Never Collected
- **File:** `src/components/doors/door-builder.tsx`
- **Description:** `FIELD_METADATA` marks `finish` as required for "all" door categories, but neither the Swing nor Slider interview paths ask for finish. Doors created through the builder will always be missing this required field.
- **Impact:** `findSpecGaps()` would flag this, but the builder bypasses gap detection.

### [M2] Door Builder: Template Path Doesn't Set `frameType`
- **File:** `src/components/doors/door-creation-flow.tsx:96-111`
- **Description:** `handleSelectTemplate` derives category, temp type, opening type, and hardware — but never sets `frameType`. For hinged doors, this is a required field.
- **Impact:** Template-created hinged doors are missing frame type on the spec sheet.

### [M3] "Confirm All" Ignores Per-Card Edits (BOM + Receiving)
- **Files:** `src/components/bom/bom-ai-flow.tsx`, `src/components/receiving/receiving-flow.tsx:87-103`
- **Description:** Both flows have a "Confirm All" button that uses original parsed quantities/costs. If the user edited values in individual confirmation cards, those edits are in local component state and are NOT reflected in the bulk confirm.
- **Impact:** Incorrect quantities/costs submitted when user edits then clicks Confirm All.

### [M4] Non-Catalog-Only Receipts Permanently Blocked
- **File:** `src/components/receiving/receipt-summary.tsx:113`
- **Description:** Submit button is `disabled={catalogItems.length === 0}`. If all confirmed items are non-catalog, the button stays disabled with no error message. Dead end.
- **Impact:** User sees items in the receipt summary but cannot submit. No explanation provided.

### [M5] Category Filter Shows Only Current Page Categories
- **File:** `src/app/inventory/page.tsx:26-34`
- **Description:** Categories are extracted from the 50 products on the current page. Categories with products only on page 2+ don't appear in the filter.
- **Impact:** Incomplete and misleading category filter.

### [M6] ProductForm Fetches Categories by Loading 500 Products
- **File:** `src/components/inventory/product-form.tsx:77-93`
- **Description:** `fetchCategories` calls `/api/inventory?limit=500` to extract unique categories. Expensive, and misses categories if there are >500 products.
- **Impact:** Performance issue + potentially missing categories in the product form dropdown.

### [M7] Assembly Detail: Shop Role Default Broken
- **File:** `src/app/assemblies/[id]/page.tsx:69-72`
- **Description:** `sheetView` is initialized from `me.role`, but `me` is undefined on first render (async). `useState` captures the initial value once and doesn't update when `me` loads. Shop floor users see the spec sheet by default instead of the manufacturing sheet.
- **Impact:** Wrong default view for DOOR_SHOP and SHOP_FOREMAN users.

### [M8] Assembly Status Filter is Dead Code
- **File:** `src/app/assemblies/page.tsx:47`
- **Description:** `statusFilter` state is declared but `setStatusFilter` is never called from any UI element. Users cannot change the filter.
- **Impact:** Dead code / incomplete feature.

### [M9] BOM Detail: Add-Material Checkout Quantities Not Populated
- **File:** `src/app/boms/[id]/page.tsx:472-484`
- **Description:** In `add-material` mode, `handleCheckout` reads from `checkoutQtys`, but `handleAIAddItems` only adds items to the BOM — it doesn't populate `checkoutQtys`. The "Confirm Checkout" button stays disabled.
- **Impact:** Confusing flow — AI-added items can't be checked out without manual quantity entry per row.

### [M10] BOM AI Flow: Duplicate Items Not Deduplicated
- **File:** `src/components/bom/bom-ai-flow.tsx`
- **Description:** Parsing the same text twice appends duplicates to `pendingMatches`. Same product appears multiple times in the BOM.
- **Impact:** Inflated material lists, wrong quantities.

### [M11] Jobs Update Schema Too Restrictive
- **File:** `src/app/api/jobs/[id]/route.ts`
- **Description:** `updateJobSchema` only allows `name`, `number`, `status`. But the Job model has 15+ fields (client, address, contractValue, dates, etc.). Any edit UI trying to update these fields gets them silently stripped.
- **Impact:** Cannot update most job fields via the API.

### [M12] Receiving Route: Partial Failure Not Transactional
- **File:** `src/app/api/receiving/route.ts:36-45`
- **Description:** Receipt items processed in a sequential loop with individual `adjustStock()` calls. If item 3 of 5 fails, items 1-2 are already committed. Receipt exists but is incomplete.
- **Impact:** Data inconsistency — partial receipts with no rollback.

### [M13] Dashboard Links to Non-Existent Inventory Filters
- **Files:** `src/components/dashboard/low-stock-list.tsx`, `src/components/dashboard/stock-summary-card.tsx`
- **Description:** Links go to `/inventory?status=low` and `/inventory?status=out`, but the inventory page doesn't read a `status` query param. Links show the full unfiltered inventory list.
- **Impact:** Misleading navigation — user expects filtered results but sees everything.

### [M14] Transaction Type Filter Unvalidated
- **File:** `src/app/api/transactions/route.ts:23`
- **Description:** `type` query param is cast as `Prisma.EnumTransactionTypeFilter` with no validation against the TransactionType enum. Invalid values cause a Prisma runtime error.
- **Impact:** API error on invalid filter values.

### [M15] Assembly Status Transitions Have No Role Checks
- **File:** `src/app/api/assemblies/[id]/route.ts`
- **Description:** Approval flow correctly requires ADMIN/SALES_MANAGER. But any authenticated user can set status to IN_PRODUCTION, COMPLETED, SHIPPED, etc. with no role restriction.
- **Impact:** Unauthorized status changes.

### [M16] `isExterior` Not Used for Hardware Defaults in AI Parser
- **File:** `src/lib/ai/parse-door-specs.ts:168`
- **Description:** Hardware lookup checks `specs.specialNotes?.toLowerCase().includes("exterior")` instead of using the `isExterior` boolean. If AI sets `isExterior: true` without mentioning "exterior" in notes, exterior hardware won't be applied.
- **Impact:** Wrong hardware defaults for exterior doors parsed by AI.

## Minor Issues (cosmetic, inconsistency, missing polish)

### [m1] Door Builder: `SWING_FRAME_CUSTOM` Not in BuilderStep Union
- **File:** `src/components/doors/door-builder.tsx:245,252`
- **Description:** Uses `as BuilderStep` cast. `STEP_GROUPS` doesn't include it, so StepProgress shows step 0 (Type) instead of step 1 (Dimensions) during the custom frame screen.

### [m2] Door Builder: `cutoutCount` is Dead State
- **File:** `src/components/doors/door-builder.tsx:68`
- **Description:** `cutoutCount` state is set but never read. `cutouts.length` is used instead.

### [m3] `parseWidthInches` Boundary at 10
- **File:** `src/lib/door-specs.ts:325-326`
- **Description:** Values <=10 treated as feet, >10 as inches. `parseWidthInches("8")` = 96" (8 feet), which is wrong for an 8-inch input. Unlikely for real door widths but a footgun.

### [m4] Fraction Notation Not Parsed
- **File:** `src/lib/door-specs.ts`
- **Description:** `parseWidthInches` doesn't handle fractions like `77-1/4"` or `3'-6"`. These pass through as NaN → 0.

### [m5] `resolveGapAnswer` Now Dead Code
- **File:** `src/lib/door-specs.ts:435`
- **Description:** After the door flow rewrite, `resolveGapAnswer` is no longer imported by any file. Also `GapQuestion` type is only used by `parse-door-specs.ts`.

### [m6] `findSpecGaps` Only Used by AI Parser
- **File:** `src/lib/door-specs.ts:257`
- **Description:** Previously used by the gap-fill UI in the old door flow. Now only consumed by `parse-door-specs.ts`.

### [m7] Inconsistent ZodError Response Formats
- **Files:** Multiple API routes
- **Description:** Some routes return `{ error: error.issues }` (array), others return `{ error: "joined string" }`. Hooks display `[object Object]` when error is an array.

### [m8] No Error Boundaries in the App
- **Description:** Zero `error.tsx` files and no `ErrorBoundary` components anywhere. A component crash shows the Next.js default error page with no recovery option.

### [m9] No `loading.tsx` Files
- **Description:** No route-level loading files. All loading states are handled inline via React Query `isLoading`, which is fine, but there's no fallback for the initial server render.

### [m10] Stock Adjust Page: No Negative Prevention
- **File:** `src/app/inventory/[id]/adjust/page.tsx:46`
- **Description:** Adjusting down below zero shows a negative preview without warning. The API might reject it, but the UI doesn't prevent it.

### [m11] Stock Adjust Page: 404 Shows "Loading"
- **File:** `src/app/inventory/[id]/adjust/page.tsx:68`
- **Description:** When `isLoading` is false and `product` is null (not found), it still shows "Loading..." instead of "Not Found."

### [m12] No Confirmation Before "Start Build"
- **File:** `src/app/assemblies/[id]/page.tsx:148-154`
- **Description:** "Start Build" immediately deducts inventory with no confirmation dialog. Unlike checkout (which confirms), this is a one-tap irreversible action.

### [m13] BOM Confirm All: Local Quantity Edits Lost
- **File:** `src/components/bom/bom-confirmation-card.tsx`
- **Description:** If user edits quantity in a card then clicks "Confirm All", the parent uses original `match.parsedItem.quantity`, not the edited value.

### [m14] Receiving: No Back from REVIEW to INPUT Preserving State
- **File:** `src/components/receiving/receiving-flow.tsx`
- **Description:** "Start over" resets everything. No way to go back and add more items without losing confirmed ones.

### [m15] Dashboard: No Error State
- **File:** `src/app/page.tsx`
- **Description:** If `useDashboard()` returns an error, the page shows "No data available" with no indication of failure or retry.

### [m16] Next.js Middleware Deprecation Warning
- **File:** `src/middleware.ts`
- **Description:** Next.js 16 has deprecated `middleware.ts` in favor of `proxy`. Non-blocking but generates a build warning.

### [m17] Cycle Count Race Condition
- **File:** `src/app/api/cycle-counts/route.ts:128-167`
- **Description:** Reading `systemQty` and calling `adjustStock()` are separate operations, not in one transaction. A concurrent stock change could make the recorded variance inaccurate.

### [m18] SVG Marker ID Collision (Theoretical)
- **File:** `src/components/doors/door-diagram-slider.tsx`
- **Description:** Uses `id="arrowhead"` for the SVG marker. Would collide if both swing and slider diagrams render on same page. Currently impossible but fragile.

## Suggestions

### [S1] Add `SWING_FRAME_CUSTOM` to the BuilderStep Union Type
Instead of casting, add it properly and include it in STEP_GROUPS.

### [S2] Create a `/api/categories` Endpoint
Replace the pattern of fetching 500 products to extract categories. Used by inventory filter and product form.

### [S3] Validate AI Responses with Zod
After `extractJSON()`, validate the shape with a Zod schema before casting. Prevents crashes from malformed AI output.

### [S4] Wrap Receiving Route in a Single Transaction
All `adjustStock()` calls in one `prisma.$transaction()` so partial failures roll back.

### [S5] Move Cycle Count Reason Input Before Submission
Or re-submit as an update after the fact.

### [S6] Add Error Boundaries
At minimum, add a root `error.tsx` that catches component crashes with a "Something went wrong" + retry button.

### [S7] Handle `status=low|out` in Inventory Page
Or change dashboard links to use a filter that the inventory page actually supports.

### [S8] Add Confirmation Dialog to "Start Build"
Since it deducts inventory irreversibly.

### [S9] Deduplicate Items in BOM and Receiving Flows
Merge duplicate products by summing quantities instead of appending.

### [S10] Use `useEffect` to Update `sheetView` Default When `me` Loads
So shop-floor users see the manufacturing sheet by default.
