# Plan: Wrap Next.js App with Capacitor for Native iOS & Android

**Created:** 2026-03-19
**Status:** Implemented
**Request:** Add Capacitor to the existing Next.js app to create native iOS and Android builds for App Store and Play Store distribution.

---

## Overview

### What This Plan Accomplishes

Adds Capacitor to the existing Next.js app so it runs as a native iOS and Android application. The web app continues to deploy on Vercel as-is — Capacitor wraps it in a native shell that loads the deployed URL, giving us App Store/Play Store presence plus access to native device APIs (push notifications, camera, haptics).

### Why This Matters

Shop floor workers need quick access to the inventory app on their phones. A native app means: one-tap launch from the home screen, push notifications for stock alerts and approvals, better camera integration for packing slip photos, and the credibility/discoverability of being in the App Store. All without rewriting a single line of the existing app.

## Design Decisions

### Key Decisions Made

1. **Server URL mode, not static export**: The app has 20+ API routes, middleware, and server-side auth. Capacitor will load `https://inventory-app-three-tan.vercel.app` in a native WebView. This means the app always shows the latest deployed version with zero native rebuild needed for web changes.

2. **iOS first, Android second**: Based on RSNE workforce device usage. iOS gets priority for initial TestFlight distribution. Android follows immediately after (same codebase, minimal extra work).

3. **Keep existing Web APIs for MVP**: The `<input type="file" accept="image/*">` already triggers the native camera picker in Capacitor's WKWebView. Web Speech API works in WKWebView. We don't need to swap to native plugins for MVP — the web APIs work fine inside Capacitor.

4. **Push notifications as first native feature**: Highest value for the shop floor — instant alerts for low stock, pending approvals, and cycle count reminders. This is the one thing the web app truly can't do well.

5. **App ID: `com.rsne.inventory`**: Standard reverse-domain naming for both platforms.

6. **Native projects committed to git**: The `ios/` and `android/` directories will be committed (minus build caches). This is Capacitor's recommendation — it allows native-level customization (splash screens, permissions, etc.).

### Alternatives Considered

- **Static export (`output: 'export'`)**: Rejected — would require rewriting all API routes as external services and breaking the integrated Next.js architecture. Massive effort for no real benefit.
- **Capacitor with local server**: Running Next.js inside the app — adds complexity, battery drain, and makes the app a server. Not worth it.
- **PWA only**: Gets home screen installation but no App Store listing, limited push notification support on iOS, and no native plugin access.
- **React Native rewrite**: Months of work to rebuild what already works. No justification.

### Open Questions

1. **Apple Developer Account**: Gabe needs an Apple Developer account ($99/year) to distribute via TestFlight and the App Store. Does RSNE already have one?
2. **Google Play Developer Account**: Google Play Console requires a one-time $25 fee. Does RSNE already have one?
3. **Push notification backend**: We'll need to store device tokens and send pushes from the server. Should this go through Supabase (which has push support) or a dedicated service like Firebase Cloud Messaging?
4. **App name for stores**: "RSNE Inventory"? Or something different?

### Step 6: Set Up Push Notifications

Create the push notification registration module and integrate it into the app layout.

**File content — `src/lib/push-notifications.ts`:**
```typescript
import { isNative } from './capacitor'

export async function initPushNotifications() {
  if (!isNative) return

  const { PushNotifications } = await import('@capacitor/push-notifications')

  const permission = await PushNotifications.requestPermissions()
  if (permission.receive !== 'granted') return

  await PushNotifications.register()

  PushNotifications.addListener('registration', (token) => {
    // Send token to our API for storage
    fetch('/api/me/push-token', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ token: token.value, platform: Capacitor.getPlatform() }),
    }).catch(console.error)
  })

  PushNotifications.addListener('pushNotificationReceived', (notification) => {
    console.log('Push received:', notification)
  })

  PushNotifications.addListener('pushNotificationActionPerformed', (action) => {
    // Handle tap on notification — navigate to relevant page
    const data = action.notification.data
    if (data?.url) {
      window.location.href = data.url
    }
  })
}
```

**Integration point:** Call `initPushNotifications()` in the root layout's client component on mount.

**Note:** The `/api/me/push-token` endpoint and push-sending infrastructure are deferred to a follow-up plan. This step wires up the client-side registration only.

**Files affected:**
- `src/lib/push-notifications.ts` (new)
